# CLAUDE.md

このファイルは、このリポジトリのコードを扱う際に Claude Code (claude.ai/code) および他のAIエージェントが参照するための単一の信頼できる情報源（Single Source of Truth）です。

> **ステータス:** 2026年サイト全面リニューアル進行中。現行の1ページLP（`index.html`）を廃止し、ハブ型サイトへ再構成する。事業背景の詳細は `IDENTITY.md` / `OFFERING.md` / `BRAND.md` / `WEBSITE.md` を参照（本ファイルと矛盾した場合は本ファイルを優先）。

## 1. プロジェクト概要

**Tech Connect** ― アプリ不要のWebARで「お客さんが足を止める体験」をつくるサービスのWebサイト。

- **最優先KPI:** 無料相談の問い合わせ件数を最大化する。すべての導線は相談フォームへ向ける。
- **今回のリニューアルで直す2つの課題:**
  - 課題A：1年前の受託制作（AR名刺・チラシ）向けの構成のままで、現在の事業（イベント向けARのレンタル、カスタマイズ済みデモを使った営業、自治体・イベント主催者へのアプローチ）とズレている。
  - 課題B：デザインに「AIっぽさ（汎用テンプレ感）」が残っている。→ 第5章のデザインルールで対策。

## 2. サイト構成（ハブ型）

現行の1ページLPを廃止し、トップ（ハブ）から各ページへ誘導する構成にする。

```
流入（検索・SNS・QR）
      ↓
トップページ（ハブ）… 実績と人柄で信頼をつくり、各所から相談へ誘導
      ├─ イベント・体験AR ページ（レンタル・出張・カスタマイズ）
      ├─ 印刷物・商品AR ページ（名刺・チラシ・CD・パッケージ）
      ├─ 事例詳細ページ（案件ごとに1本ずつ追加できる資産）
      └─ AR入門コラム（旧「ARとは？」等の教育コンテンツの移転先）
      ↓
無料相談フォーム（全ページ共通の出口）
```

### URL設計・リダイレクト方針
- トップは `/`（現行と同じ）を維持する。
- 新ページ例：`/event-ar/`（イベント・体験AR）、`/print-ar/`（印刷物・商品AR）、`/works/<案件名>/`（事例詳細）、`/column/<記事名>/`（コラム）。
- ページ分割で旧URLを変える場合は、リポジトリに `netlify.toml`（または `_redirects`）を新規追加し、旧→新を 301（恒久的な転送）で繋ぐ。
- **【確認済み・低リスク】** 名刺・チラシに印刷済みのQRコード、および「AR体験する」ボタンの遷移先は、**このサイトとは別デプロイのAR体験URL**。よってサイトのページ分割でQRが切れる心配はない。ただし公開前チェックでは必ず実機で再確認する。

## 4. デザインルール（脱・AIっぽさ）

「AIっぽさ」＝均等なカードグリッドの羅列・絵文字の多用・汎用ワード・イラスト/アイコン中心の構成。以下をルール化する。

- **実写の写真・動画を主役**にする。イラスト・汎用アイコンは脇役に留める。
- **絵文字は一切使わない。**
- 同じ大きさのカードを等間隔に並べる構成を避け、サイズ・配置に**強弱**をつける。
- **色数を絞る**（ベース＋アクセント1色が基本）。
- 見出しに「〜とは？」「◯つのメリット」型を使わない。原稿で確定した見出し文言をそのまま使う。
- **CTAボタンは主・副の2種類のみ**。視覚的な強さに明確な差をつける。
- **モバイルファースト**（流入はQR・Instagram経由が中心でスマホ閲覧が大半）。

### デザイントークン（CSS `:root`、現行 `style.css` の実値）
| 用途 | 変数 | 値 |
|------|------|----|
| メイン（紺） | `--navy` | `#1c2f56`（ブランド正式色。現行CSSは `#1a2a4a` のため、リニューアル実装時に更新する） |
| 紺（明） | `--navy-light` | `#2B4570` |
| アクセント（黄・全体の1割まで） | `--accent-yellow` | `#ffed00` |
| ベース（薄灰） | `--light-gray` | `#F8F9FA` |
| 文字（濃灰） | `--dark-gray` | `#333333` |

> **紺色は `#1c2f56`（ブランド正式色）に統一すると決定。** 現行 `style.css` は `#1a2a4a` なので、リニューアル実装時に `--navy` を `#1c2f56` に更新すること。

## 5. コピー・メッセージング原則

- **キーワードは「体験」。** メインコピーは「お客さんが足を止める「体験」、つくります。」。「体験」は素のまま通すのが原則で、「感動」「驚き」は1〜2箇所だけ添える。
- NG言い換え：「サプライズ体験」「常識を覆す体験」（広告的な誇張＝AIっぽさの典型）。
- **ARという手段ではなく、お客さん側に起きる結果**（足を止める・記憶に残る・盛り上がる）を先に語る。
- 想定読者はAR知識ゼロのイベント主催者・店舗オーナー。**専門用語を避け、ですます調**で書く。
- 「ARとは？」「STEP解説」「メリット列挙」などの教育・型コンテンツはトップから外し、コラム・FAQへ移す。
- 相談ハードルを下げる言葉を各所に置く（「売り込みはしません」「思いつきの段階でOK」「断ってください」）。

## 6. サービス構成（顧客目的軸の2本柱＋相談枠）

技術ではなく「顧客が何をしたいか」で分類する。

- **柱1：イベント・体験AR**（マルシェ・商業施設・展示会・地域イベント向け）。提供形態は関わりの深さ順に3段階：
  1. 既製レンタル（完成済みコンテンツをそのまま貸出。URL納品、運営は主催者側）
  2. カスタマイズレンタル（ロゴ・素材の差し替え。短時間で完了する開発体制が強み。**主力商品**）
  3. 企画・出張サポート（新規制作、当日の設営・運営同行まで）
- **柱2：印刷物・商品AR**（名刺・チラシ・カタログ・パッケージ・CD向け）。AR CD はここに含める（仕組み・顧客目的が名刺・チラシと同じため）。
- **相談枠（カスタム）:** 上記に当てはまらないもの（観光マップ・文化財演出・オリジナル企画）。独立した柱にはしない（実績の薄さを目立たせないため）。**実績が1件できたら3本目の柱に昇格**させる方針。

## 9. 開発・制作ワークフロー

### ローカル開発
- サーバー起動：`python3 -m http.server 8000` → http://localhost:8000/
- CSS最適化が必要な場合のみ：`npx tailwindcss -i style.css -o style.bundle.css --watch`（開発中はCDN版を使用）

### 制作フロー（全体）
```
STEP1 素材の棚卸し（実写動画・写真・数字・声）
STEP2 コピー確定 … 完了（成果物：トップページ原稿）
STEP3 デザインシステム設定（色・フォント・コンポーネント）
STEP4 Claude Design でモックアップ制作（FVから1セクションずつ）
STEP5 handoff bundle を Claude Code に渡して実装 ←★ Claude Code の担当
STEP6 公開前チェック＋計測設定（GA4でフォーム送信をCV計測）
```

### コーディング規約
- **HTML/JS**: 4スペースインデント／**CSS**: 2スペースインデント
- 命名：顧客向けの言葉に沿った説明的なクラス名（例：`.hero-text-enhanced`）
- 文字コード：既存のUTF-8（日本語テキスト/ファイル名）は保持する

## 12. 作業の進め方・品質基準（AIエージェント向け）

### 12.1 品質バーとレビュー
ページを公開する前に、次の3層のレビューを通すこと。
1. **セルフレビュー:** 実装直後に自分で見直す（壊れ・重複・置き去りのプレースホルダ）。
2. **自動レビュー:** `/code-review`（バグ・簡素化）と `/security-review`（セキュリティ）を実行する。
3. **2視点クロスレビュー:** 下記サブエージェント2体に並行レビューを依頼し、指摘を統合して反映する。

### 12.2 セキュリティ（このプロジェクトの範囲）
静的サイト＋Netlifyフォームのため攻撃面は小さいが、以下は必ず守る。
- 秘密情報（APIキー・トークン等）をリポジトリにコミットしない。`.env` 等は `.gitignore`。
- 外部リンクは `rel="noopener noreferrer"`。
- フォームはNetlifyのハニーポット対策を維持。個人情報の扱いはプライバシーポリシーと整合させる。
- 第三者のCDN/スクリプトは信頼できるものに限定。可能なら SRI（`integrity`）を付ける。
- フォーム等のJSでユーザー入力を `innerHTML` 等で危険に展開しない。
- 依存（Three.js 等）はバージョンを固定する。

### 12.3 トークン効率・実装計画（本当の節約はここ）
- **1ページ／1セクション単位**で進める。一度に全部作らない。
- 大きなファイルは**必要な箇所だけ読む**（全文を何度も読み直さない）。
- **デザインシステム（色・余白・コンポーネント）と共通部品（ヘッダー/フッター/フォーム）を使い回す**。毎回ゼロから作らない。
- 既存実装（フォーム・パーティクル）は流用する。
- サブエージェントは**合計トークンを減らす道具ではない**。品質と並行作業のための道具であり、要所でのみ使う（小さな修正・文言変更には使わない）。

### 12.4 サブエージェント運用（2体のレビュー体制）
`.claude/agents/` に2体を用意済み。いずれも**読み取り専用（自分では変更しない）**で、指摘・改善案を返す。適用はメインの担当が行う。
- **シニアエンジニア（`senior-engineer`）:** 正確性・パフォーマンス・モバイル堅牢性・アクセシビリティ・セキュリティ・保守性。
- **プロデザイナー（`pro-designer`）:** 視覚的階層・脱AIっぽさ・ブランド整合（色/フォント/余白）・余白とリズム・実写主役。

**使いどころ:**
- ① ページが一通り完成したら、2体を**並行で**走らせてクロスレビュー → 指摘を統合して反映。
- ② 独立した複数セクション/ページがある場合の分担制作（同じファイルを同時に編集させないこと）。

**使わない場面:** 1〜数行の小修正、文言だけの変更。

> 注：2体は現在「レビュー専門（読み取り専用）」。エージェント自身にコードを書かせたい場合は、各定義ファイルの `tools` に `Edit, Write` を追加する。
